[ 32. 테스트 계획과 시나리오 ]
	1. 테스트 종류
		1) 단위/통합테스트
			- 사용자가 서비스를 이용하는 시나리오로 테스트
		2) 성능테스트
			- 로딩 시간, 정확한 정보 출력 등의 성능
		3) 스트레스(부하)테스트
			- 동접 등의 부하
		4) 벤치마킹테스트
			- 벤치마킹 사이트를 테스트이용해보며 기획에 반영
		5) A/B테스트
			- 기획 케이스를 나눠 테스트해 더 효율적인 기획을 선택
	
	2. 테스트 계획
		1) 일정
			- 단위/통합테스트 차시별 계획 수립
			- 테스트 대상자 스케줄 및 개발 완료 상태 확인
		2) 대상
			- 현업 테스트 대상자 선정 및 일정 통보
			- 제2차 사용성 테스트 대상자 리크루팅
			- 테스트 일정 및 프로세스 사전 공유
			- 테스트 진행 인력을 수행 TFT에서 선별
		3) 장소
			- 테스트 대상자 일정을 고려한 장소 섭외
			- 수행 TFT와 별도의 공간 마련
		4) 범위
			- 기능별 단위테스트 범위 결정
			- 통합테스트 차시별 범위 및 조건 결정
			- 테스트 제외 사항에 대한 의사결정
		5) 환경
			- 하드웨어 소프트웨어 네트워크 환경에 따른 테스트 필요 여부를 판단하여 차시별 다른 환경 조건에서 통합 테스트 진행
		6) 시나리오
			- 수행 TFT 단위/통합테스트 시나리오 준비
			- 시나리오 수행을 위한 데이터 및 권한 계정 준비
			- 시나리오는 반드시 각 PL들의 검수 후 배포
		7) 디버깅
			- 테스트 후 Fail 항목들에 대한 처리 일정 협의
			- Fail 항목에 대한 디버깅 및 예외사항 피드백
			- 디버깅 완료 후 테스트를 통한 최종 확인
	
	3. 시나리오 작성
		- 사용자 방문 목적 중심의 테스트 시나리오를 작성할 것
		예시) 인터넷 뱅킹 홈페이지
		방문목적	|	목적 달성을 위한 사용자 행동 시나리오
		--------|-------------------------------------

[ 33. 테스트 후 처리해야 할 문제 ]
	1. 오류 체크리스트
		1) 오류내용
			- 오류 내용은 구현 단계에서 커뮤니케이션한 용어를 반드시 사용하여 최대한 상세히 작성하고, 캡쳐나 관련 URL 정보를 제공함으로써 오류 처리 담당자가 재현하여 정확한 현상을 파악하여 처리하도록 기술
		2) 구분
			- 오류, 개선, 변경, 추가로 구분하여 '오류'를 최우선으로 처리하고 수행 TFT에서 가용한 스케줄을 협의하여 오류→개선→변경→추가의 순서로 처리를 진행
			- 오류 외에 처리되지 않은 개선, 변경, 추가 사항에 대해서는 각 수행 TFT PL들의 처리 방안에 대한 의견을 담아 운영팀에 이관하여 진행 가능하도록 지원
			* 테스트결과를 오류, 개선, 변경, 추가 사항으로 구분은 수행사와 현업간에 이견이 발생하는 경우가 많음. 그러므로 요구사항분석 단계에서의 요구사항정의서 및 구현단계의 화면설계서, 회의록 등의 증빙문서 작성이 매우 중요한 순간
			* 누락 사항은 논외, 최우선의 별도 이슈
		3) 처리 결과
			- 처리 결과는 처리중, 확인 필요, 반려로 구분하여 표기하고, 처리 완료가 되면 완료 일자를 표기하며, 반여릐 경우에는 반려사유를 반드시 명시하여 의사결정권자의 확인 후 클로징
		4) 의견
			- 의견은 반려사유를 포함하여 각 오류처리 담당자와 현업의 요청자 사이에 오류 처리에 대한 이견 히스토리를 관리하기 위해 작성하며, 작성일과 작성자를 반드시 명시하여 추후에 미스커뮤니케이션이 일어나지 않도록 관리
	
	2. 사용자 예외 처리 테스트
		1) error page
			- 기본적인 에러페이지 외에 비 정상접근 등의 사용자 오류 대응 가이드를 할 수 있는 안내페이지 마련
		2) alert message
			- 요청사항에 대한 처리결과 피드백 뿐만 아니라, 고관려 의사결정 시나 취소가 불가능한 결정 시 사용자 오류를 범할 수 있는 사항을 재확인하는 단계를 마련
	

[ 34. 최종 산출물 정리와 메뉴얼 정리 ]
	1. 오픈 시나리오
		- 내부 오픈 일정 및 범위와 grand open 일정 및 이벤트 진행에 대한 시나리오 수립
		1) 운영 서버 이관 시나리오
			- 서버의 시스템환경을 파악하여 개발 서버에서 운영 서버로 이관하는 일정 수립
		2) 롤백 시나리오
			- 운영 서버 이관이나 grand open 시 예기치 않은 오류가 발생할 경우 이전 시스템으로 롤백할 담당자 선정, 업무 분장, 롤백 순서, 일정, 롤백 후 체크리스트 등 시나리오 수립
		3) 오픈 모니터링 시나리오
			- 오픈 후 서비스 모니터링 및 CS접수에 대한 실시간 대응 계획(상황별 운영) 및 트래픽과 사용 로그 분석을 위한 시나리오 수립
		4) 장애 대응 시나리오
			- 프로젝트 철수 후 긴급장애 발생 시 대응 인력과 비상연락망 확보 및 프로세스에 대한 시나리오 협의
	
	2. 최종 상출물
		1) PM/기획
			- 착수보고서 / 업무명세서(WBS) / ★요구사항정의서 / 조직구성도 / 마스터스케줄 / ★정보구조도 / 기능정의서 / 수행기획한 / 화면설계정의서 / 커뮤니케이션 가이드 / ★화면설계서 / 테스트결과서 / 주간보고&회의록 / ★사용자매뉴얼 / ★운영자매뉴얼 / 검수확인서
		2) 디자인
			- 디자인 컨셉서 / 디자인 스타일 가이드 / 네이티브 가이드 / 디자인 시퀀스 / 디자인 원본 파일
		3) 퍼블리싱
			- 퍼블리싱 가이드 / 스크립트 원본파일 / 퍼블리싱 원본파일
		4) 프로그래밍
			- 디렉토리 구조도 / ERD / 엔티티 정의서 / 테이블 정의서 / 프로그램 목록 / 프로그램 사양서 / 프로그램 원본 파일
	
	3. 사용자/운영자 매뉴얼
		1) 사용자 매뉴얼
			- 고객센터의 CS 응대를 위해 배포용으로 제작되어 MS Office로 만들어 PDF로 배포하거나, 플래시로 별도 제작하여 홈페이지 상에서 제공
		2) 운영자 매뉴얼
			- 운영팀이 서비스 운영 및 관리를 위해 관리자 페이지 이용가이드 뿐만 아니라, 디자인/퍼블리싱/프로그래밍의 운영 매뉴얼을 제공
	
	4. 내부리뷰 AAR
		- 프로젝트 오나료 후 각 파트별 수행 TFT 및 협업 동료간 업무수행 과정과 결과에 대한 자체 리뷰
		- 문제점과 개선방향뿐 아니라, 진행 중에 경험한 리스크와 그 대응방안에 대해 리뷰를 진행함으로써 타 프로젝트에서의 효과적인 대응방안에 대한 정보를 공유하고, 본 프로젝트의 기획/디자인/퍼블리싱/프로그래밍상의 테크니컬한 장점과 아쉬운 부분을 리뷰하여 추후에 사전 리스크 체크 및 유용한 기술 활용을 위한 취지

[ 35. 프로젝트 제안 필수 사항 ]
	1. 제안요청서 RFP를 완전히 숙지하자
	2. 제안목차를 먼저 정리하라
	3. 이야기의 흐름을 만들어라
		- 사업의 목적, 목적을 달성하기 위한 목표, 목표를 이루기 위한 전략, 전략을 수행하기위한 전술
	4. 디자인 시안을 위한 프로토타입을 설계하라
	5. 구체적인 실행안을 제시하라
		- 제약사항과 전제사항을 명확히 공유하고
		- 예산과 일정 안에 수행 가능한 범위 내에서 최선을 제안한다
